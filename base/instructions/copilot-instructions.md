## Правила рефакторинга для всех агентов

### Импорты при перемещении файлов
При конвертации `part of` → `import` или перемещении файлов между директориями:
1. Относительные пути к `core/`, `app/`, `l10n/` пересчитывать от НОВОГО расположения файла.
2. Не копировать импорты из старого `part`-файла — они были привязаны к родительскому файлу.
3. После каждого перемещения/конвертации запускать `flutter analyze`.
4. Если ошибка `uri_does_not_exist` — значит путь на один `../` короче или длиннее, чем нужно.
5. Проверять, что все публичные классы/функции, используемые в других файлах, импортируются явно.

### Проверка PM
- PM обязан проверять `flutter analyze` после каждого рефакторинга.
- Если PM делегировал задачу — запросить у субагента результат `flutter analyze` и убедиться, что **0 errors**.
- При внесении изменений в код: делай минимальные и целенаправленные правки, соблюдай стиль проекта, при необходимости обновляй тесты и документацию.
- Код писать в минималистичном стиле: без лишних оберток и без избыточных fallback-структур.
- Реализацию вести структурированно и модульно, соблюдая единый стиль и существующую архитектуру каталогов/подкаталогов.
- Для каждого нового или измененного модуля, класса и метода добавлять docstring на русском языке.
- Комментарии в коде писать только по делу и только на русском языке.
- Не придумывать поведение и структуры: опираться на документацию и текущую реализацию репозитория.
- При расхождении документации и текущей реализации обязательно задавать уточняющие вопросы до внесения изменений в код.
- Автодеплой запрещен: любые действия по деплою выполнять только после явного согласия заказчика.
- **SCP/SSH/docker cp на сервер запрещены**: изменения кода доставляются только через git push → GitHub Actions. Прямое копирование файлов или пересоздание контейнеров вручную недопустимо (кроме аварийного восстановления).

Контактные заметки для ассистента:

- Перед любым изменением проверяй существующие тесты и конфигурации в `server/tests/`.
- При крупных изменениях сначала предложи план действий и используй todo-лист для отслеживания прогресса.

## 🏗 АРХИТЕКТУРНЫЙ КОНТРАКТ ДЛЯ ИИ

Все ИИ-ассистенты обязаны соблюдать отдельный серверный контракт:

- `docs/server_docs/server_architecture_ai_contract.md`

Кратко:
- backend следует трёхслойной архитектуре `route -> service -> infrastructure`;
- route не должен выполнять SQL напрямую;
- service принимает `uow`, а не `session`;
- repository отвечает только за доступ к данным;
- при изменениях сервера нужно проверять соответствие документации в `docs/server_docs/` и обновлять тесты.
